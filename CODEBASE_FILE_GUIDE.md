# OCRPro 代码库文件作用总览（给 Claude / AI 助手快速定位用）

> 目标：让 AI 在修改代码前，先看这份文档就能快速知道“每个文件是做什么的、该改哪里、不要改哪里”，减少全仓库重复解析。

---

## 1. 项目整体架构（一句话）

这是一个 **Python + pywebview + 前端 JS 表格** 的询价/价目匹配工具：
- 前端负责录入、展示、编辑、复制导出；
- `api.py` 作为 JS 与 Python 的桥接层；
- 后端核心模块完成 OCR、RFQ 解析、数据库查询、描述匹配；
- SQLite (`database_data.db`) 存储 FullList 价目数据。

## 2.1 根目录 Python / 启动 / 构建相关

### `app.py`
- **角色**：桌面应用主入口（启动 pywebview 窗口）。
- **核心职责**：
  - 配置日志（写入 `app.log`）；
  - 定位前端入口 `frontend/index.html`；
  - 创建窗口并注入 `API` 对象（`js_api`）；
  - 启动 webview 事件循环。
- **何时修改**：
  - 想改窗口尺寸、标题、启动行为；
  - 想调整是否开启 debug/devtools。
- **不要在这里做的事**：业务逻辑（应放 `api.py` / 业务模块）。

### `api.py`
- **角色**：前后端桥接层（最核心调度层）。
- **核心职责**：
  - 暴露给前端 JS 调用的方法（配置加载、OCR、查询、RFQ 解析、导出等）；
  - 调用 `database.py` 完成数据查询；
  - 调用 `matcher.py` 完成描述相似匹配；
  - 调用 `ocr_engine.py` 完成 OCR 抽取；
  - 维护价目表缓存，减少重复查库。
- **何时修改**：
  - 前端新增按钮/动作，需要新增 Python API；
  - 需要调整“描述匹配优先、代码匹配兜底”等流程；
  - 需要联动多个后端模块的编排。
- **注意**：此文件最容易成为“巨石层”，建议新功能拆成独立模块后由 `api.py` 调用。

### `config.py`
- **角色**：全局配置与字段映射单一来源。
- **核心职责**：
  - 数据库路径解析（开发/打包双模式）；
  - FullList 数据库列名与展示列名映射（`FL_DB_COLS`, `FL_DISPLAY`）；
  - 价格列、公司列索引常量；
  - 公司下拉选项、列宽配置。
- **何时修改**：
  - Excel/数据库字段变化；
  - 公司字段变化；
  - 列显示名、列宽策略变化。
- **高风险点**：列顺序必须与数据库一致，改错会导致整表错位。

### `database.py`
- **角色**：数据库访问层（DAL）。
- **核心职责**：
  - 连接 SQLite；
  - 读取 FullList（按公司返回对应价格列）；
  - 单品查询（IMPA/U8/模糊兜底）；
  - 批量查询；
  - 全库关键字搜索。
- **何时修改**：
  - 需要新增数据库查询接口；
  - 需要调整 SQL、查询优先级、返回字段。
- **边界建议**：其他模块尽量不要绕过它直接访问 DB。

### `matcher.py`
- **角色**：描述语义匹配引擎。
- **核心职责**：
  - 两阶段匹配：先类目路由再语义相似计算；
  - 支持 BGE ONNX（可用时）与 TF-IDF（兜底）双模式；
  - 维护缓存与 top-k 检索结果。
- **何时修改**：
  - 提升描述匹配准确率；
  - 调整相似度阈值、分词策略、候选集策略；
  - 切换/优化嵌入模型。
- **定位提示**：如果“匹配错品类”或“同义词识别差”，优先看这里 + `category_router.py`。

### `category_router.py`
- **角色**：描述文本的规则类目路由器。
- **核心职责**：
  - 通过关键词/正则将查询归类到领域（例如电气、机械、密封等）；
  - 用于缩小 `matcher.py` 的检索候选范围，减少跨类误匹配。
- **何时修改**：
  - 新增业务品类；
  - 某类目召回率/准确率明显偏低；
  - 需要增加排除词（negative patterns）。

### `ocr_engine.py`
- **角色**：OCR 引擎与屏幕截取逻辑。
- **核心职责**：
  - 处理 Tesseract 路径（开发/打包）；
  - 执行 OCR 文本抽取；
  - 从 OCR 文本解析 item code、描述、数量、单位等结构化字段。
- **何时修改**：
  - OCR 识别率差；
  - 新增代码格式/单位规则；
  - 截图交互流程调整。
- **高风险点**：本机路径依赖（Tesseract）和正则策略耦合较强。

### `Rfq_quotation_tool.py`
- **角色**：RFQ 页面解析器（URL/本地 HTML）。
- **核心职责**：
  - 抓取或读取 HTML；
  - 识别询价表格并抽取标准字段（Code/Description/Qty/UOM）；
  - 提供给 `api.py` 调用的 `parse_rfq_url()` 接口；
  - 同时保留 CLI 独立模式。
- **何时修改**：
  - SevenSeas 页面结构变化导致解析失败；
  - 需要兼容新的 RFQ 模板来源。

### `DatabaseUpdate.py`
- **角色**：Excel → SQLite 导入工具（含 GUI）。
- **核心职责**：
  - 读取 Excel 首行作为字段；
  - 重建 FullList 表并批量导入；
  - 提供导入进度与状态回调；
  - 提供独立 Tk 窗口用于手动更新数据库。
- **何时修改**：
  - 导入字段规则调整；
  - 导入性能问题；
  - 想加数据校验/预清洗。

### `databaseCheck.py`
- **角色**：数据库健康检查脚本（调试工具）。
- **核心职责**：
  - 打印数据库路径、大小；
  - 列出所有表的字段与行数。
- **何时修改**：
  - 需要更多诊断信息（索引、空值率等）。

### `UMIHOSHI.spec`
- **角色**：PyInstaller 打包配置文件。
- **核心职责**：
  - 定义入口脚本、资源文件、隐藏依赖、输出模式。
- **何时修改**：
  - 打包资源缺失；
  - 依赖无法被正确收集；
  - 需要调整 onedir/onefile 等策略。

### `build.bat`
- **角色**：Windows 一键构建脚本。
- **核心职责**：
  - 触发构建流程（通常调用 PyInstaller）；
  - 做简单的清理或拷贝。
- **何时修改**：
  - CI/CD 或本地打包步骤变化。

### `__init__.py`
- **角色**：包标记文件（当前为空）。
- **核心职责**：
  - 让根目录可被 Python 视作包（视运行方式而定）。
- **何时修改**：
  - 需要暴露统一版本信息/包级工具函数时。

## 2.2 前端资源（`frontend/`）

### `frontend/index.html`
- **角色**：前端主页面骨架。
- **核心职责**：
  - 定义布局容器、按钮、面板、弹窗结构；
  - 引入样式和脚本（`style.css`, `js/main.js`, `js/grid.js`）。
- **何时修改**：
  - 新增/调整 UI 结构；
  - 增加新的交互入口控件。

### `frontend/style.css`
- **角色**：全局样式。
- **核心职责**：
  - 页面布局、配色、组件视觉样式；
  - 表格可读性（间距、行高、状态色）。
- **何时修改**：
  - 视觉改版；
  - 控件样式、响应式细节优化。

### `frontend/js/main.js`
- **角色**：前端应用状态与业务交互主控（Alpine.js 组件）。
- **核心职责**：
  - 管理全局状态（语言、公司、tab、loading 等）；
  - 处理用户操作（OCR、查询、匹配、导出、RFQ 粘贴解析）；
  - 调用 `window.pywebview.api.*` 与后端通信；
  - 管理 i18n 文案、提示信息。
- **何时修改**：
  - 新增按钮行为/流程；
  - 交互时序优化；
  - 多语言文案扩展。

### `frontend/js/grid.js`
- **角色**：表格渲染与交互逻辑（网格层）。
- **核心职责**：
  - 处理列显示、行高、单元格编辑、选中、排序/筛选（若有）；
  - 承担数据视图层更新与部分操作手势。
- **何时修改**：
  - 表格体验问题（卡顿、编辑行为、列逻辑）；
  - 新增表格维度功能（例如批量编辑、固定列）。

## 3. 按“改动目标”快速定位

### 3.1 想改“查询匹配逻辑”
优先看：
1. `api.py`（匹配编排、调用顺序）
2. `matcher.py`（语义算法）
3. `category_router.py`（类目过滤）
4. `database.py`（查库字段与 SQL）

### 3.2 想改“OCR 识别效果”
优先看：
1. `ocr_engine.py`（识别与解析）
2. `frontend/js/main.js`（OCR 调用触发与结果落表）

### 3.3 想改“RFQ 链接解析”
优先看：
1. `Rfq_quotation_tool.py`（HTML 解析核心）
2. `api.py`（前后端接口联动）
3. `frontend/js/main.js`（粘贴链接交互）

### 3.4 想改“前端页面/按钮/表格体验”
优先看：
1. `frontend/index.html`（结构）
2. `frontend/style.css`（样式）
3. `frontend/js/main.js`（动作）
4. `frontend/js/grid.js`（表格行为）

### 3.5 想改“数据字段/公司列/列宽”
优先看：
1. `config.py`（全局配置源）
2. `database.py`（字段映射使用）
3. `api.py`（返回列组合逻辑）
4. `frontend/js/main.js`（列显示与语言映射）

### 3.6 想改“导入新 Excel 到数据库”
优先看：
1. `DatabaseUpdate.py`
2. `database_data.db`（只验证结果，不直接手改）
3. `databaseCheck.py`（检查导入是否生效）

## 4. 建议 Claude 修改代码时的最小阅读顺序

当任务是“常规功能改动”时，建议先读：
1. 本文档（`CODEBASE_FILE_GUIDE.md`）
2. `config.py`（字段与全局常量）
3. `api.py`（调用主链）
4. 与任务直接相关的 1~2 个模块（如 `matcher.py` / `main.js`）

这样通常可以在不全仓扫描的情况下完成定位。
